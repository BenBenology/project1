# Agent 协作说明

这个文件定义项目内 AI Agent 或协作者的工作边界，目的是让后续继续开发时保持节奏一致、结构一致。

## Agent 目标

Agent 的目标是帮助团队持续推进这个项目，但必须遵守 MVP 优先原则：

- 优先打通主流程，不提前过度工程化
- 优先保留模块边界，便于未来替换前端和接入真实爬虫
- 优先输出可运行结果，而不是只输出想法

## 当前项目阶段

当前阶段是 `MVP 骨架验证阶段`，重点是：

- 跑通查询任务主链路
- 保持前后端分层
- 用 mock 数据代替真实抓取
- 为接入真实数据源留出接口

## Agent 工作原则

### 1. 先保结构，再加能力

任何新增功能都应尽量落在明确分层中：

- 路由层只做入参处理和响应组织
- 服务层负责业务流程
- 仓储层负责数据读写
- mock 逻辑与真实抓取逻辑分离

### 2. 不破坏前后端解耦

即使当前前端是 Streamlit，也不能把业务逻辑写到前端里。前端只消费 API，不直接依赖后端内部实现。

### 3. 优先替换点清晰

后续最容易变化的部分有三块：

- 数据源抓取逻辑
- 数据存储方式
- 前端表现层

Agent 在改动时要尽量保证这些部分可以被单独替换。

### 4. 避免提前引入重型依赖

如果当前需求只需要 mock 和最小演示，不要提前引入复杂消息队列、ORM、大型前端框架，除非任务明确要求。

## 推荐工作顺序

1. 先补文档
2. 再补接口和模型
3. 再替换 mock 为真实实现
4. 最后再做优化、缓存、调度、监控

## 禁止事项

- 不要把“自动炒股”理解成自动交易系统
- 不要默认抓取所有站点全文并长期存储
- 不要把临时 demo 逻辑扩散到整个代码库
- 不要让 Streamlit 直接承担后端职责

## 输出要求

Agent 交付内容默认应包含：

- 修改了什么
- 为什么这样拆分
- 当前仍有哪些限制
- 下一步最合理的推进方向
